Problem Statement

Current OPD Challenges

  • Waiting Room Congestion: Patients spend hours in crowded halls with no visibility on wait times.
  • Unstructured Queues: Verbal or paper tokens lead to skipping, VIP biases, and unfairness.
  • Revenue Leakage: Manual billing between hospitals and doctors causes accounting errors.
  • Absence of Data: No digitized patient medical history or e-prescriptions.

Traditional Tier-2 and Tier-3 outpatient departments suffer from severe operational inefficiencies.

The NOQ Solution

NOQ is a role-based digital ecosystem that transforms hospital queues into seamless, real-time workflows.

Live Tracking

Patients monitor their token status remotely to minimize physical wait times.

Fair FIFO Queueing

Strict chronological token processing with zero VIP bypass capabilities.

Digital Records

Integrated EMR and e-prescriptions instantly accessible to doctors.

Technology Stack

  • Frontend: React 19, Vite, Tailwind CSS
    Component-driven UI for fast, dynamic state rendering.
  • Backend API: Python FastAPI
    Asynchronous (ASGI) processing for high concurrency and Pydantic validation.
  • Database: MongoDB
    NoSQL document store, perfect for dynamic clinical records and high-speed atomic transactions.

Why we migrated from Firebase to FastAPI

Initial prototype used Firebase, which caused massive per-read costs for analytics. Moving to a custom FastAPI + MongoDB stack reduced API response times to ~45ms and allowed complex backend aggregations.

4-Tier RBAC Architecture

The system completely segregates endpoints and dashboards to protect medical privacy.

1. Platform Admin

System oversight, audits, and manager approvals.

2. Hospital Manager

Manages doctors, rooms, and local daily queues.

3. Doctor

Consults patients and issues e-prescriptions.

4. Patient

Books tokens and views medical history.

Workflow: Booking Lifecycle

  • Initial Selection

    Patient selects the preferred hospital branch and specific disease/specialty.

  • Smart Allocation

    System algorithm matches the patient to the most available doctor and room for that specialty.

  • Secure Payment

    Patient pays the required consultation fee and confirms their appointment securely.

  • Geofence & Token Generation

    System tracks location depending on the booking type, then assigns a strict sequential token number.

Geofencing & Booking Types

Normal Booking

Normal token booking requires the patient to be physically present within the hospital's surroundings (e.g. 500m radius). The algorithm tracks location via GPS to manage hospital traffic accurately based on time.

Advanced Booking

Bypasses location tracking, allowing patients to book from home. Strictly reserved for vulnerable demographics to prevent queue manipulation:

  • Pregnancy cases
  • Infants and babies (<= 7 years)
  • Elderly patients (> 70 years)

Atomic Token Generation

Simultaneous booking requests create race conditions. NOQ uses database-level atomicity to prevent duplicate tokens. 


async def generate_token(db, queue_id):
    # MongoDB executes this as a single atomic operation
    queue = await db.queues.find_one_and_update(
        {"_id": queue_id},
        {"$inc": {
            "token_counter": 1, 
            "waiting_count": 1
        }},
        return_document=ReturnDocument.AFTER
    )
    return queue["token_counter"]

* The $inc operator safely increments the counter without requiring a standard read-then-write pattern, guaranteeing token uniqueness.

Revenue Split & System Ethics

Automated Revenue Split

  • When an appointment is marked as completed, the backend automatically splits the fee:
  • 80% to the Hospital Manager
  • 15% to the consulting Doctor
  • 5% platform service fee
  • Eliminates manual accounting errors.

Queue Responsibility

  • No VIP Pass: Strict FIFO calling is enforced on the doctor's dashboard.
  • Penalty System: If a patient is consistently absent when called, an automated ₹50 penalty fee is added to their next booking to prevent queue abandonment.
