Project & Architecture Overview

What is NOQ?

NOQ is an enterprise-grade, role-based healthcare workflow optimization platform. It replaces paper-based registries and archaic verbal tokens with a real-time digital queue system connecting Platform Admins, Hospital Managers, Doctors, and Patients.

Architectural Evolution & Migration

The project was originally prototyped on Google Firebase. However, real-world benchmarking exposed critical performance bottlenecks and high costs due to per-document read charges (especially for analytics dashboards).

The Solution: Migrated to a custom backend stack using FastAPI (Python) for async request processing and MongoDB. This allowed for database-side aggregations, dropping API response times from ~400ms to ~45ms and improving dashboard load speeds by 80%.

4-Tier User Hierarchy (RBAC)

1. Platform Admin

  • Oversight: Approves/rejects new Hospital Manager registrations.
  • Analytics: Monitors the global 5% platform service fee.
  • Compliance: Full access to system-wide audit logs.

2. Hospital Manager

  • Administration: Manages specific hospital branch locations.
  • Resources: CRUD operations on Doctors, Rooms, and Departments.
  • Queues: Configures daily OPD queues and capacities.

3. Doctor

  • OPD Interface: Sequential 'Call Next' queue processing.
  • Clinical Records: Writes digital prescriptions and tracks refills.
  • History: Accesses patient past medical history instantly.

4. Patient

  • Booking: Geofenced live tokens or advanced appointments.
  • Live Tracking: Monitors token position remotely to avoid waiting rooms.
  • Wallet & Feedback: Pays fees, downloads EMRs, and files complaints.

Core Logic & Algorithms

Geofenced vs. At-Home Booking Policy

Normal Tokens (Geofenced): Must be booked within physical proximity of the hospital (e.g., 500 meters) to prevent spam. The backend computes the Haversine distance between patient GPS and hospital coordinates.

Advanced Bookings (At-Home): Permitted ONLY for 3 vulnerable groups to prevent queue manipulation:

  • Pregnancy Cases
  • Infants & Children (under 7 years)
  • Elderly Patients (70+ years)

Atomic Token Generation

To prevent race conditions (two patients getting the same token), NOQ bypasses standard read-then-write logic. It uses MongoDB's $inc operator inside a single find_one_and_update command. This ensures engine-level atomic locks for perfectly sequential token numbers.

Platform Equality

There are zero VIP overrides or fast-tracks. The system forces strict FIFO (First-In-First-Out) calling based purely on check-in timestamps.

Revenue Split

Automated backend hooks instantly divide consultation fees into an 80% (Hospital), 15% (Doctor), and 5% (Platform) split upon completion.
